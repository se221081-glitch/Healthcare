This table documents 5 key design decisions where the web and mobile implementations differ, with WCAG rationale for each choice.

Platform comparison: 5 key web vs mobile design decisions
Design Decision Web App (Desktop) Mobile App WCAG Rationale
Navigation Pattern Persistent left sidebar (260px) with icon + label nav items Bottom tab bar with 4 primary destinations WCAG 2.4.1: Multiple navigation mechanisms. Sidebar persistent on desktop (large target area, always visible); bottom tab reachable by thumb on mobile.
Date Input Native <input type="date"> with keyboard support and visible text fallback instructions Native iOS/Android date picker wheel with haptic feedback WCAG 2.1.1 Keyboard: Desktop requires full keyboard access. Mobile platform pickers provide native accessibility (VoiceOver/TalkBack) out of the box.
Modal Presentation Centred overlay with backdrop, focus trap, Escape to close Full-screen sheet sliding up from bottom (native feel) WCAG 2.4.3 Focus Order: Desktop overlay must trap focus within bounds. Mobile full-screen sheets are simpler to navigate for screen reader users on small displays.
Form Layout 2-column grid on 1280px, single column on 768px and below Single column always; larger tap targets (min 48x48dp per Material) WCAG 1.3.4 Orientation + 2.5.5 Target Size: Responsive web adapts via CSS Grid. Mobile enforces single column and larger targets due to finger-based input.
Data Visualisation CSS bar charts with visible labels, text summary paragraph, and togglable data table Native chart library with pinch-to-zoom, with accessible text summary always shown below WCAG 1.1.1 Non-text Content: Both platforms always pair charts with text summaries. Web additionally offers a data table toggle. Mobile relies on platform accessibility API for gesture descriptions.
