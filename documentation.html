Colour Accessibility Token Table

All colours are defined as CSS custom properties in tokens.css. Contrast ratios are verified against WCAG 1.4.3 (text) and 1.4.11 (non-text). Every status colour is paired with an icon and a text label — colour is never the sole means of conveying information (WCAG 1.4.1).

Primary Colour Justification

Teal (#218080) was selected as the primary brand colour because it sits between blue and green on the spectrum, making it distinguishable under both Deuteranopia (red-green deficiency) and Tritanopia (blue-yellow deficiency) — unlike pure green or pure red which collapse under common colour-blindness conditions. At shade 600, it achieves a 4.69:1 contrast ratio on white (WCAG AA) and scales to AAA at shade 700 and above, ensuring every interactive state meets or exceeds the minimum threshold without requiring a separate high-contrast theme. Its association with health, calm, and clinical trust reinforces the application domain without relying on culturally loaded colours such as red (danger) or yellow (caution), which are reserved exclusively for the semantic Error and Warning tokens.

Colour-blindness simulation note

The chosen teal palette was reviewed against common colour-vision conditions, including Deuteranopia and Tritanopia. The semantic status colours are paired with icons and labels so that meaning is preserved even when colour perception differs.

WCAG 2.1 Compliance Document

The following tables document every WCAG 2.1 criterion addressed, organised by screen. Each entry identifies the criterion, describes the specific design/code decision taken, and states the conformance level achieved.

All Screens (Global)

Criterion Description Implementation Level
1.1.1 Non-text Content All decorative SVG icons have aria-hidden="true" focusable="false". Informative icons have adjacent visible text labels. The steps chart uses role="img" with a full aria-label describing all data points. AA
1.4.3 Contrast (Minimum) Body text (neutral-700) achieves 10.1:1 on white. All interactive text uses primary-600 at 4.69:1 minimum. Helper text uses neutral-500 at 4.6:1. No text falls below 4.5:1. AA
1.4.4 Resize Text All font sizes use rem units relative to the browser root. No text is set in fixed px. Text reflows correctly at 200% browser zoom. AA
1.4.12 Text Spacing Line height set to 1.5 (WCAG minimum 1.5×). Letter-spacing tokens defined (0.02em–0.05em). No content is clipped when text spacing is overridden via user stylesheet. AA
2.1.1 Keyboard All interactive elements (buttons, links, form fields, sliders, modals) are reachable and operable via keyboard. No keyboard traps except the intentional focus trap inside modals (WCAG 2.1.2 compliant — Escape closes). AA
2.3.3 Animation from Interactions @media (prefers-reduced-motion: reduce) block in tokens.css disables all CSS transitions and animations globally. Toast fade, section slide-in, and card hover lift are all suppressed. AAA
2.4.1 Bypass Blocks Every page has a visually hidden skip link (<a href="#main-content">) that becomes visible on focus. Styled with primary-700 background for high contrast. AA
2.4.7 Focus Visible CSS custom property --focus-ring: 0 0 0 3px #FFF, 0 0 0 5px #2B9C9A applied to all interactive elements via :focus-visible. Double-ring design ensures visibility on any background colour. AA
2.5.3 Label in Name All button and link visible text exactly matches their accessible name. aria-label values that override visible text include the visible text as a substring. AA
2.5.5 Target Size --target-min: 2.75rem (44px) applied via min-height: var(--target-min) on all buttons, links, form inputs, and checkboxes. Mood selector buttons also enforce min-width: 44px. AA
3.1.1 Language of Page lang="en" is set on the <html> element of every page. AA
4.1.3 Status Messages Toast notification region has aria-live="polite" aria-atomic="true". Form error summary uses aria-live="assertive". Range slider value outputs have aria-live="polite". AA

Criterion Description Implementation Level

Platform Design Comparison — Web vs Mobile

Five key design decisions where the web and mobile implementations intentionally differ, with the WCAG rationale for each choice.

Design Decision Web App (Desktop 1280px) Mobile App (iOS/Android) WCAG Rationale
Navigation Pattern Persistent left sidebar (260px) with icon + text labels. Always visible, no toggle required. Bottom tab bar with 4 primary destinations, reachable by thumb in natural hand position. WCAG 2.4.1 Multiple ways to navigate. Sidebar persistent on desktop provides always-visible context. Bottom tab reachable by thumb on mobile reduces motor effort for elderly users (WCAG 2.5.5).
Date Input Native <input type="date"> with keyboard support. Fallback manual text entry instructions provided below the field. Native iOS/Android date picker wheel with haptic feedback and VoiceOver/TalkBack integration. WCAG 2.1.1 Keyboard access. Desktop requires full keyboard operability. Mobile platform pickers provide native accessibility APIs (announce selected value to screen readers) out of the box without custom ARIA.
Modal Presentation Centred overlay with semi-transparent backdrop, programmatic focus trap (trapFocus()), Escape key to close. Full-screen bottom sheet sliding up — native feel, simpler focus model on small displays. WCAG 2.4.3 Focus Order. Desktop overlay must programmatically trap focus within its bounds to prevent screen reader users from interacting with obscured content. Mobile full-screen sheets are naturally bounded by the viewport.
Form Layout 2-column CSS Grid on 1280px; collapses to single column below 640px. Labels above inputs. Single column always. Minimum 48×48dp touch targets (Material Design) — larger than web's 44px minimum. WCAG 1.3.4 Orientation + 2.5.5 Target Size. Responsive CSS Grid adapts web layout. Mobile enforces larger targets due to finger-based input and variable grip positions of elderly users.
Data Visualisation CSS bar charts with value labels, always-visible text summary, and a togglable data table alternative. Native chart component with pinch-to-zoom; accessible text summary always displayed below the chart. WCAG 1.1.1 Non-text Content. Both platforms always pair charts with text summaries. Web additionally offers a data table toggle (keyboard accessible) for users who need raw data. Mobile relies on platform accessibility API for gesture descriptions.
